# Locations of interest — design

**Issue:** [#85 — Locations of interest: configurable list of places with per-apartment distances](https://github.com/brlauuu/flatpare/issues/85)
**Follows up:** [#56 — single configurable train station](https://github.com/brlauuu/flatpare/issues/56)
**Date:** 2026-04-26

## Problem

#56 shipped a single configurable train-station address. Bike/transit minutes for every apartment are computed from that single origin. The user wants up to 5 named "locations of interest" (work, partner, gym, etc.) with per-apartment distances to each, plus an icon-based UI for picking and displaying them.

## Scope

- New `locations_of_interest` table (max 5 rows, app-wide).
- New `apartment_distances` join table — bike + transit minutes per (apartment × location).
- Drop `apartments.distance_bike_min` / `distance_transit_min` columns. Migrate existing values into the join table under a default location ("Train Station", icon `Train`, address from the existing `app_settings.station_address`).
- Drop `app_settings` table entirely (no other consumer remains).
- `/settings` page: list of locations with add / edit / delete, max 5 enforced, mobile-friendly reorder via up/down arrows, address field, icon picked from a curated 12-lucide-icon modal grid.
- Apartment detail: one row per location with the chosen icon. Clicking the icon opens Google Maps directions from apartment → location with `travelmode=bicycling`.
- Compare view: one row per location, **icon-only header** with `title` tooltip showing the label. Each cell renders `bike / transit` minutes inline.
- Apartments list sort: dynamic per-location options, `Bike to {label}` / `Transit to {label}`.
- Recompute-all button: re-runs every (apartment × location) pair.
- Manual distance form fields on apartment edit are dropped — recompute is the only way to populate distances.

## Storage

```ts
// src/lib/db/schema.ts
export const locationsOfInterest = sqliteTable("locations_of_interest", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  label: text("label").notNull(),
  icon: text("icon").notNull(), // lucide name; validated against ICON_NAMES
  address: text("address").notNull(),
  sortOrder: integer("sort_order").notNull(),
  createdAt: integer("created_at", { mode: "timestamp" }).default(sql`(unixepoch())`),
  updatedAt: integer("updated_at", { mode: "timestamp" }).default(sql`(unixepoch())`),
});

export const apartmentDistances = sqliteTable(
  "apartment_distances",
  {
    apartmentId: integer("apartment_id").notNull().references(() => apartments.id, { onDelete: "cascade" }),
    locationId: integer("location_id").notNull().references(() => locationsOfInterest.id, { onDelete: "cascade" }),
    bikeMin: integer("bike_min"),
    transitMin: integer("transit_min"),
    updatedAt: integer("updated_at", { mode: "timestamp" }).default(sql`(unixepoch())`),
  },
  (t) => [primaryKey({ columns: [t.apartmentId, t.locationId] })]
);
```

The `apartments` table loses `distance_bike_min` / `distance_transit_min`. The `app_settings` table is dropped.

### Migration (single file)

Generated by `db:generate`, then manually augmented to include the backfill. Order:

1. `CREATE TABLE locations_of_interest (...)` and `apartment_distances (...)`.
2. `INSERT INTO locations_of_interest (label, icon, address, sort_order) SELECT 'Train Station', 'Train', value, 0 FROM app_settings WHERE key = 'station_address';`
3. Fallback if no `app_settings` row: `INSERT INTO locations_of_interest (label, icon, address, sort_order) SELECT 'Train Station', 'Train', 'Basel SBB, Switzerland', 0 WHERE NOT EXISTS (SELECT 1 FROM locations_of_interest);`
4. `INSERT INTO apartment_distances (apartment_id, location_id, bike_min, transit_min) SELECT a.id, (SELECT id FROM locations_of_interest WHERE sort_order = 0 LIMIT 1), a.distance_bike_min, a.distance_transit_min FROM apartments a WHERE a.distance_bike_min IS NOT NULL OR a.distance_transit_min IS NOT NULL;`
5. SQLite-style `ALTER TABLE apartments DROP COLUMN distance_bike_min;` and `DROP COLUMN distance_transit_min;` (SQLite ≥ 3.35).
6. `DROP TABLE app_settings;`

## Distance lib

`src/lib/distance.ts`:

```ts
export interface DistanceResult { bikeMinutes: number | null; transitMinutes: number | null; }
export async function calculateDistance(originAddress: string, destinationAddress: string): Promise<DistanceResult>;
```

The single-station read is removed. Callers that previously called `calculateDistance(apartmentAddress)` must now provide the origin (a location address) and the destination (the apartment address). Same Google Maps and ORS branches as before, just parameterized.

`src/lib/app-settings.ts` is deleted (no remaining consumers after the migration).

## API routes

### `GET /api/locations`

Returns `Location[]` ordered by `sort_order`.

### `POST /api/locations`

Body: `{ label, icon, address }`. 400 if any field invalid or already 5 rows. Returns the created location with computed `sortOrder = max + 1`.

### `GET /api/locations/[id]`, `PUT /api/locations/[id]`, `DELETE /api/locations/[id]`

Standard CRUD. PUT body accepts partial `{ label?, icon?, address? }`. DELETE cascades into `apartment_distances`.

### `POST /api/locations/[id]/move`

Body: `{ direction: "up" | "down" }`. Swaps sort orders with the adjacent row.

### `POST /api/settings/recompute-distances`

Reused name, generalised behaviour. Iterates `apartments × locations`. For each pair where the apartment has an address, calls `calculateDistance(location.address, apartment.address)` and upserts into `apartment_distances`. Response: `{ totalApartments, totalLocations, updated, failed, skipped }` where `skipped` counts (apartment, location) pairs where the apartment has no address.

The old `GET /api/settings` and `PUT /api/settings` routes are deleted.

## Apartment payload changes

- `GET /api/apartments` (list): no longer returns `distanceBikeMin`/`distanceTransitMin`. Adds `distances: { locationId: number; bikeMin: number | null; transitMin: number | null }[]`.
- `GET /api/apartments/[id]` (detail): same.
- `POST /api/apartments` (create): no longer accepts distance fields. After insert, computes distances for every configured location and inserts rows into `apartment_distances`. (Same flow used by both manual create and PDF parse.)
- `PATCH /api/apartments/[id]` (edit): no longer accepts distance fields. Distances are unchanged unless recompute is triggered.

### Icon-grid modal

`src/components/location-icon-picker.tsx` — a 4×3 grid of all 12 icons. Each icon is a button; clicking one selects it and closes the dialog.

### Apartment detail

```
... existing badges ...

[Train] Train Station   12 min bike · 18 min transit
[Briefcase] Work         8 min bike · 14 min transit
```

- The icon is wrapped in an `<a target="_blank" href="https://www.google.com/maps/dir/?api=1&origin={apartmentAddress}&destination={locationAddress}&travelmode=bicycling">`.
- If a distance value is null, render `—` instead of `N min`.
- If there are zero locations configured, the entire section is hidden.

### Compare view

Replace the two hardcoded distance rows in `metricRows` with one row per location. Row header: icon-only with `title="Bike + transit to {label}"`. Each cell shows `12 / 18 min` (or `12 / —`, `— / —`). Best-value highlighting is applied per-mode independently — too noisy when summed; instead highlight the row's best `bike+transit` total. Acceptable simpler choice for v1: skip best-value highlighting on these rows.

### Apartments list sort

Sort options become dynamic:

- Static fields stay (`createdAt`, `rentChf`, `sizeM2`, `numRooms`, `avgOverall`, `shortCode`).
- For each location L: `bikeTo:<L.id>` and `transitTo:<L.id>` with labels `Bike to {label}` / `Transit to {label}`.

The persisted sort field may reference a deleted location id; the validator falls back to `createdAt` in that case (acceptable — single-user app).

`SortableApartment` gains `distances: Map<number, { bikeMin: number | null; transitMin: number | null }>`. Extractors look up the per-location entry by id.

Compare view sort uses the same superset.

## Out of scope

- Per-user sets of locations.
- Per-apartment manual distance overrides.
- Drag-to-reorder.
- Caching geocoded coords across providers.
- Streaming progress for the recompute endpoint.
- Best-value highlighting on the per-location rows in the compare view.

## Risks

- **SQLite drop-column** requires SQLite ≥ 3.35 (libsql is built on a recent SQLite, so this is safe).
- **Recompute API budget**: 5 locations × N apartments × 2 modes is up to 10× the previous. Personal app, low N — acceptable.
- **Compare-view legibility**: 5 icon-only rows can blend together; the tooltip is the safety net.
- **Stored sort field becomes invalid** after a location is deleted; the validator handles it by falling back to default.
